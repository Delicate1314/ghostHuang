# LLM/VLM 增强语义 ID 的电影生成式推荐 — 项目说明报告

---

## 1. Project Topic（项目课题）

### 1.1 研究题目

**基于 LLM/VLM 增强语义 ID 的电影生成式推荐研究**
（LLM/VLM-Enhanced Semantic ID for Generative Movie Recommendation）

### 1.2 问题定义

传统推荐系统使用随机整数 ID（如 `movie_3847`）表示物品，这种 ID 不携带任何语义信息。当新电影加入系统时，由于缺乏用户交互数据，模型无法为其生成有效推荐——这就是经典的**冷启动问题**。

我们的核心研究问题是：

> **如何利用 LLM/VLM 对电影进行多模态语义建模，构建语义增强的离散 ID（Semantic ID），并通过生成式推荐范式提升推荐效果，尤其在新电影冷启动场景下？**

### 1.3 技术路线概述

我们设计了一个 6 模块的端到端框架：

| 模块 | 功能 | 技术选型 |
|------|------|----------|
| 数据构建 | MovieLens 1M + LLM 世界知识补全元数据 | DeepSeek API 异步并发 |
| LLM 语义建模 | 提取 9 维结构化语义标签（genre_fine, mood, theme, pace, ...） | DeepSeek-V3.2 API |
| VLM 视觉建模 | 海报视觉风格描述 → CLIP ViT-L/14 编码 | CLIP 文本编码器 |
| 多模态对齐 | InfoNCE 对比学习，融合文本+视觉表示与协同过滤信号 | PyTorch, 用户共现对 |
| 语义 ID 构造 | 显式语义 token + 层次化 K-Means 隐式量化 token | scikit-learn |
| 生成式推荐 | Transformer Decoder 自回归预测下一部电影的 SID | PyTorch |

### 1.4 为什么选电影

电影天然是多模态对象（文本、图像、结构化属性、行为序列），非常适合展示 LLM + VLM + Semantic ID + 冷启动的综合能力。

### 1.5 创新点

1. **LLM 驱动的语义中间层**：不让 LLM 直接做推荐，而是作为"语义增强器"，从世界知识中提取高层属性（mood, theme, narrative_style 等）
2. **融合视觉信息的多模态 Semantic ID**：将海报视觉描述与文本语义对齐后共同构造 SID，使 SID 包含视觉风格信号
3. **面向冷启动的生成式推荐**：新电影无需交互历史，仅凭 LLM/VLM 提取的语义即可构造 SID 参与推荐

## 2. Progress（已完成的工作）

### 2.1 数据构建（已完成 ✅）

**MovieLens 1M 数据集**：
- 下载并解析 1,000,209 条评分记录
- 3,883 部电影，6,040 个用户

**用户行为序列构建**：
- 正反馈过滤（rating ≥ 4）
- 按时间排序，Leave-last-one-out 划分
- 得到 6,034 条有效用户序列，平均长度 37.7

**冷启动划分**：
- 随机选取 15% 电影（555 部）作为"新电影"
- 剩余 3,328 部为"老电影"

**LLM 世界知识补全元数据**：
- 由于 TMDb API 被网络屏蔽，改用 DeepSeek API 的世界知识直接生成每部电影的：overview（剧情简介）、directors（导演）、cast（演员）、poster_description（海报视觉描述）
- 使用 `asyncio + httpx + Semaphore(15)` 异步并发，速度 3.3 部/秒
- 结果：3,882/3,883 部成功（99.97%），输出 `movies_meta.parquet`

### 2.2 LLM 语义标签提取（已完成 ✅）

使用 DeepSeek-V3.2 API 对 3,883 部电影提取 9 维结构化语义标签：

```json
{
  "genre_fine": ["buddy-comedy", "adventure-comedy", "family-adventure"],
  "mood": "whimsical",
  "theme": ["friendship", "jealousy", "homecoming"],
  "pace": "moderate",
  "audience": "family",
  "narrative_style": "linear",
  "visual_style": "colorful",
  "era_setting": "modern",
  "emotion_arc": "jealousy to friendship adventure"
}
```

- 异步并发提取，3,882/3,883 部成功
- 输出 `llm_semantic_tags.json`

### 2.3 文本嵌入生成（已完成 ✅）

- 使用 **BGE-M3**（BAAI/bge-m3）将结构化语义标签编码为 1024 维向量
- 输入格式：`"Genre: X. Mood: Y. Theme: Z. Pace: W. ..."`
- 输出：`text_embeddings.npy` — shape `(3882, 1024)`

### 2.4 视觉代理特征提取（已完成 ✅）

由于 TMDb 被屏蔽无法下载真实海报图片，我们采用**视觉代理方案**：

- 用 DeepSeek 世界知识生成的 `poster_description`（如 "The poster likely features Woody and Buzz Lightyear prominently against a playful, colorful background..."）
- 通过 **CLIP ViT-L/14 文本编码器**编码为 768 维向量
- 输出：`clip_embeddings.npy` — shape `(3883, 768)`

**这比简单跳过视觉模态更好**：poster_description 包含色调、构图、氛围等视觉语义，CLIP 文本编码器保留了与视觉空间的对齐能力。

### 2.5 多模态对齐训练（已完成 ✅）

- 从用户序列构建 500,000 个共现电影对（窗口=5）
- 使用 **InfoNCE 对比学习**对齐文本（1024d）和视觉（768d）表示
- 投影到共同的 256 维空间，加权融合（α=0.6 偏向文本）
- 训练 50 epochs，Loss 从 5.64 降至 5.53
- 输出：`fused_embeddings.npy` — shape `(3882, 256)`

### 2.6 语义 ID 构建（已完成 ✅）

采用**混合型 Semantic ID**设计：

**显式部分（9 个 token）**：
- 3 个 genre_fine token（如 `buddy-comedy`, `adventure-comedy`, `family-adventure`）
- 6 个属性 token（mood, pace, audience, narrative_style, visual_style, era_setting）
- 词表大小：1,810

**隐式部分（3 个 token）**：
- 对融合嵌入做 3 级层次化 K-Means（每级 32 聚类，残差量化）
- 词表大小：96

**最终 Semantic ID**：每部电影 12 个 token，总词表 1,906

示例：`Movie 1 → [8, 9, 10, 2, 3, 4, 5, 6, 7, 1812, 1871, 1877]`

### 2.7 推荐模型训练（已完成 ✅）

- **模型架构**：Transformer Decoder，d_model=128, nhead=4, num_layers=2, max_movies=10
- **参数量**：1,022,708（约 1M）
- 双层位置编码（电影级 + SID 内部级）
- 自回归训练：预测下一个 SID token
- **训练配置**：batch_size=64, lr=1e-3, CosineAnnealingLR, 20 epochs
- **训练结果**：Loss 从 3.55 降至 0.60（CPU 纯训练约 1.5 小时）
- 模型保存至 `checkpoints/sid_model.pt`

### 2.9 模型评估（已完成 ✅）

采用自回归生成 + 加权 SID 匹配的评估方案：

1. 给定用户历史序列，模型自回归生成 12 个 SID token
2. 将生成的 SID 与所有电影的 SID 做**加权 token 匹配**（genre 权重 3, attribute 权重 2, implicit 权重 1）
3. 取 Top-10 候选，计算 Recall@10, NDCG@10, MRR

**评估结果：**

| 指标 | Overall（5,055 样本） | Cold-Start（979 样本） |
|------|:--------------------:|:---------------------:|
| Recall@10 | 0.0318 | 0.0296 |
| NDCG@10 | 0.0201 | **0.0211** |
| MRR | 0.0166 | **0.0186** |
| HitRate@10 | 0.0318 | 0.0296 |

**关键发现：冷启动电影的 NDCG@10（0.0211）和 MRR（0.0186）均高于整体平均水平**，表明基于 LLM/VLM 语义构造的 Semantic ID 能够有效缓解冷启动问题——即便电影从未出现在训练交互中，模型仍可通过语义 ID 的匹配将其推荐给合适的用户。

### 2.8 数据产出汇总

| 文件 | 说明 | 大小 |
|------|------|------|
| `movies_meta.parquet` | 电影元数据（LLM补全） | 3,883 部 |
| `llm_semantic_tags.json` | 结构化语义标签 | 3,882 部 |
| `text_embeddings.npy` | BGE-M3 文本嵌入 | (3882, 1024) |

### 3.2 遇到的问题

**问题 1：TMDb API 网络不通**
- 公司/校园网限制，无法访问 api.themoviedb.org
- **已解决**：改用 DeepSeek API 的世界知识直接生成电影元数据。这反而更贴合课题"利用 LLM 世界知识增强推荐"的主旨

**问题 2：HuggingFace 模型下载困难**
- CLIP ViT-L/14 (1.7GB) 下载多次超时
- **已解决**：使用 HF 镜像 + `sentence-transformers` 封装加载，BGE-M3 虽慢但最终成功；CLIP 通过 open_clip 的 OpenAI 预训练权重下载

**问题 3：无 GPU 可用**
- 当前设备只有 Intel 集显，无 NVIDIA 独显
- **影响**：模型训练纯 CPU，速度较慢（BGE-M3 推理 50 分钟，Transformer 训练预估 1-2 小时）
- **已缓解**：缩小模型（d_model 128, 2 层, max_movies 10），参数量从 5.2M 降至 1M

**问题 4：多线程 API 死锁**
- ThreadPoolExecutor 并发请求 DeepSeek API 时，在 199/3883 处卡死
- **已解决**：改用 `asyncio + httpx + Semaphore(15)` 异步方案，稳定且快（3.3 req/s）

**问题 5：Python 兼容性**
- 系统 Python 3.14 不支持 PyTorch
- **已解决**：用 `uv` 安装 Python 3.12，创建独立 venv

### 3.3 当前困难

1. **CPU 训练效率低**：Transformer Decoder 训练每个 epoch 约 4-5 分钟，20 epochs 共约 1.5 小时，限制了超参搜索的效率。模型已从 5.2M 缩减至 1M 参数以适应 CPU
2. **无真实海报图片**：只能用文本描述作为视觉代理，可能弱化了 VLM 模块的贡献
3. ~~**评估流程待搭建**~~：✅ 已完成——实现了自回归 SID 生成 + 加权 token 匹配方案，6,034 用户全量评估约 20 分钟

### 3.4 希望获得的建议

1. 在纯 CPU 环境下，是否有更高效的轻量模型架构推荐？
2. 用文本描述代替真实海报图片做 CLIP 编码，在论文中如何更好地论证其合理性？
3. 消融实验的设计：去掉显式 vs 隐式 token、去掉视觉特征、去掉 LLM 标签——这几个对比够吗？

## 4. Potential Solutions & Milestone Plan

### 4.1 剩余问题的解决方案

**推荐模型评估方案：**
- 给定测试集中的用户历史序列，模型自回归生成 12 个 SID token
- 将生成的 SID 与所有电影的 SID 做余弦相似度匹配 → Top-K 候选
- 计算 Recall@10, NDCG@10, HitRate@10, MRR

**冷启动实验方案：**
- 将 555 部冷启动电影从训练交互中完全移除
- 但保留它们的 Semantic ID（仅由 LLM/VLM 语义构造）
- 测试模型能否将这些"从未见过交互"的电影推荐给用户
- 单独报告 Cold-Recall@10, Cold-NDCG@10

**消融实验矩阵：**

| 变体 | 显式 token | 隐式 token | 视觉特征 | LLM 标签 |
|------|:----------:|:----------:|:--------:|:--------:|
| Full Model | ✓ | ✓ | ✓ | ✓ |
| w/o Explicit | ✗ | ✓ | ✓ | ✓ |
| w/o Implicit | ✓ | ✗ | ✓ | ✓ |
| w/o Visual | ✓ | ✓ | ✗ | ✓ |
| w/o LLM | ✓ | ✓ | ✓ | ✗ |
| Random ID | ✗ | ✗ | ✗ | ✗ |

**基线方法：**
- Pop（热门推荐）
- BPR-MF（矩阵分解）
- SASRec（序列推荐 SOTA）
- 使用 RecBole 框架快速跑基线
